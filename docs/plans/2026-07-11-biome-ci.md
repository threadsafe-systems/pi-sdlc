# Plan: biome + CI (GitHub Actions)

- Date: 2026-07-11
- Track: **reversible** (fast path). Tooling — the iron law's own text:
  "Everything else is reversible (internal refactors, docs, tests,
  tooling)." No schema, no public contract, no frozen surface.
- Split from `docs/plans/2026-07-11-semantic-release.md` (v1, since renamed)
  after plan-panel review: two of three vendors independently caught that
  bundling this reversible tooling work with the genuinely irreversible
  release pipeline was inconsistent with this project's own
  track-classification rule. See
  `docs/reviews/plan-ci-biome-semantic-release-2026-07-11/consolidated.md`.
- Brief brainstorm: this session's design conversation with the project
  owner; approved to proceed directly to plan.

## Objectives

1. **Biome** (minimal scope): consistent, unambiguous formatting, and a
   structural safety net for JSON/config files.
2. **CI** (GitHub Actions, first ever in this repo): run `npm test` and a
   biome check on every PR — and make it an actual enforced gate, not just a
   workflow file that exists (see the honest gap noted below).

## Rationale

Zero prior lint/format tooling, zero CI, on a codebase now big enough that a
structural mistake is a real cost, not a hypothetical one — caught during
this session's own development: while fixing a PR-panel finding on the
in-flight model-thinking-levels change, an edit put a literal newline inside
a JSON string (invalid JSON), caught only because `python3 -m json.tool` was
run manually before committing. A JSON-aware linter in CI would have caught
it automatically, before a human had to notice.

## An honest gap, named plainly (from plan-panel review)

A `.github/workflows/ci.yml` file existing does **not** make it an enforced
gate — GitHub will happily let a PR merge with a red or unrun check unless a
branch-protection rule requires that check to pass. This plan scopes in that
one-time repository-settings step explicitly, rather than implying the
workflow file alone is sufficient.

## Scope

### In

- `.github/workflows/ci.yml`: on PR to `main` — `npm ci`, `npm test`, biome
  check (format + lint, verify-only, never auto-fix in CI).
- `@biomejs/biome` devDependency + minimal config: formatter matching this
  repo's existing convention (tabs, double quotes — confirmed by
  inspection) plus a conservative/recommended-safe lint ruleset, not the
  full strict set, so first adoption doesn't require triaging a pile of
  pre-existing findings. Exact version and config keys pinned at spec time.
- `package.json`: `lint`/`format` scripts wrapping biome (check vs. write
  modes respectively).
- **Branch protection**: a required-status-check rule on `main` requiring
  the CI workflow to pass before merge — a one-time GitHub repository
  setting, documented with the exact `gh api`/UI step at spec time, not
  something committed to the git tree but still part of this plan's DoD.

### Out

- The semantic-release pipeline (own plan, irreversible track).
- Full/strict biome linting (deferred; minimal now).
- Any commit-message-format checking (that lives in the release-pipeline
  plan, since it's specifically in service of that pipeline's correctness).

## A cross-plan tension — RESOLVED by the project owner

(Superseded; kept for history.) The tension identified after the second
PR-panel round — direct-push openness vs. closing the CI-bypass gap — is
resolved. The project owner replaced classic branch protection with a
GitHub **ruleset** requiring one approving PR review on `main`, bypassable
only by `OrganizationAdmin` (their own account). The general CI-bypass gap
this section originally flagged is closed by that ruleset. The sibling
semantic-release plan's dependency on a direct push is satisfied
separately: a fine-grained PAT (`RELEASE_PAT`, scoped to `Contents: Read
and write` on this repo only, owned by the bypassed account) authenticates
that one specific automated push — see
`docs/specs/2026-07-11-semantic-release.md` §2.3 for the full mechanism.
No remaining tension between the two plans.

## Definition of done

- [ ] CI workflow present and green on this PR.
- [ ] Branch protection on `main` actually requires the CI check to pass
      (verified, not just configured — a PR with a deliberately broken test
      or biome violation must be blocked from merging).
- [ ] Biome added; `npm run lint`/`format` scripts present; minimal ruleset
      chosen and documented.
- [ ] `npm test` green; no regression to the 33 existing tests.
